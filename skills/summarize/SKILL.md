# 总结与提炼

当前项目主要依靠 LLM、`web_fetch`、文件工具、MCP 搜索/网页工具和已注入的对话上下文完成总结，不依赖外部 `summarize` CLI。

## 触发场景

- 用户发来长文本，希望总结重点。
- 用户发来 URL，希望了解网页内容。
- 用户要求“提炼要点”“TL;DR”“概括一下”“总结这篇文章/对话/资料”。
- 用户要求整理 workspace 文件、RSS/feed 条目、搜索结果或近期对话。
- Drift 空闲时整理网页、feed、记忆摘要或长期上下文，但只产出对用户有用的内容。

## 工具策略

1. 如果输入是普通文本，直接总结。
2. 如果输入是 URL：
   - 优先用 `web_fetch(url)` 获取正文。
   - 如果需要更完整网页抓取，用 `tool_search(query="fetch page web content")` 查找 MCP 网页工具。
   - 抓取失败时说明限制，不要编造网页内容。
3. 如果是 workspace 文件：
   - 必要时先用 `list_files(path)` 找文件。
   - 再用 `read_file(path, max_chars)` 读取；文件太长时先总结可读部分，并说明截断。
4. 如果是最新信息、新闻、价格、天气：
   - 先用可见搜索/MCP 工具获取来源，再总结。
   - 如果搜索后端降级为空结果，明确说明未查到可靠来源，不要用旧知识补实时结论。
5. 如果是图片或附件：
   - 只有视觉模型启用且上下文中已有图片内容时，才能总结图片。
   - 如果只能看到附件类型摘要，不要假装读到了图片细节。

## 输出格式

默认使用简洁中文：

```text
一句话结论：...

要点：
1. ...
2. ...
3. ...

值得注意：...
```

如果用户要求非常短，只输出 1-3 条要点。用户要求详细时，再补充背景、证据和不确定处。

如果是主动 drift 输出，第一行必须是适合 Telegram/QQ 主动推送的自然短句，不要写成后台报告、任务日志或“以下是摘要”。后续归档内容可以使用 Markdown。

## 注意

- 区分“来自工具的内容”和“模型推断”。
- 不要输出大段原文；除非用户要求摘录，也只给短引用。
- 来源不可靠、抓取失败或内容被截断时，要明确说明。
- 保留数字、日期、人物、结论和行动项；删除寒暄、重复和无关细节。
- 不要把总结写成营销文案，除非用户明确要求改写。
